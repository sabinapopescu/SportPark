# SportPark

A group-training scheduling and registration app for a fitness club (Fitness, CrossFit, Pilates, Tenis, Squash, Kids Park). Visitors browse the upcoming training schedule and reserve a spot with just their name — no account required. Club staff manage the schedule and rosters through a lightweight admin panel.

The UI text is in Romanian (`ro`); this document describes the code in English.

## Tech stack

| Layer | Choice |
|---|---|
| Framework | [TanStack Start](https://tanstack.com/start) (SSR) on top of [TanStack Router](https://tanstack.com/router) (file-based routing) |
| UI | React 19 + Tailwind CSS v4 + [shadcn/ui](https://ui.shadcn.com) (new-york style, Radix primitives) |
| Data fetching | [TanStack Query](https://tanstack.com/query), wired into router loaders via `@tanstack/react-router-with-query` for SSR |
| Backend | [TanStack Start server functions](https://tanstack.com/start/latest/docs/framework/react/guide/server-functions) (RPC-style, same app — no separate API server) |
| Database | MySQL (MariaDB via XAMPP in dev) + [Prisma ORM](https://www.prisma.io) |
| Auth | Session cookie (HttpOnly), bcrypt password hashing |
| Build | Vite 8, `@tanstack/router-plugin` for route generation |
| Lint/format | ESLint (flat config) + Prettier |
| Package manager | `bun` (lockfile committed); `npm` also works |

## Getting started

1. **Database.** Start MySQL (e.g. via XAMPP's control panel or `mysql_start.bat`), then create the database once:
   ```sql
   CREATE DATABASE sportpark CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
   ```
2. **Env vars.** Copy `.env.example` to `.env` and fill in `DATABASE_URL` (defaults to `mysql://root@localhost:3306/sportpark`, matching a stock XAMPP install) and a fresh `ENCRYPTION_KEY` (command included in the file).
3. **Install, migrate, seed:**
   ```bash
   npm install            # or: bun install
   npm run db:migrate     # applies prisma/migrations, generates the Prisma client
   npm run db:seed        # sample events + a demo admin account (prints the login)
   ```
4. **Run it:**
   ```bash
   npm run dev             # http://localhost:5173
   npm run build            # production build -> dist/
   npm run preview          # preview the production build
   npm run lint / format
   npm run db:studio        # Prisma Studio, a GUI for the DB
   ```

## Features / modules

### Public schedule — `src/routes/index.tsx`
The home page (`/`). Lists upcoming training events grouped by three tabs — **Azi** (today), **Mâine** (tomorrow), **Săptămâna aceasta** (this week) — sorted by start time. Each event card shows its category badge, banner image, occupied/total seats, and either a live registration countdown, "seats full", or "registration closed" state. Cards link to the event detail page unless registration is disabled.

### Event detail & registration — `src/routes/event.$id.tsx`
The `/event/$id` page. Shows full event info (description, date/time, location, coach) and a registration form (name required, phone optional) with:
- A honeypot field for basic bot/spam protection (checked again server-side).
- Client-side validation (name length, phone length), re-enforced server-side.
- A live 1-second countdown to the registration deadline.
- A server-side rejection (event full / registration closed) if capacity or the deadline changed between page load and submit — the frontend doesn't trust its own snapshot.

On success it shows a confirmation with a **personal cancellation link** (`/cancel/$token`) that can be copied to the clipboard, plus a list of other registered attendees (names only — no phone numbers ever reach the public page — shown as first name + last initial for privacy; the current user sees their own full name).

### Cancel registration — `src/routes/cancel.$token.tsx`
The `/cancel/$token` page. Looks up a registration by its unique cancellation token and lets the attendee confirm cancellation, freeing their seat. Handles invalid tokens and already-cancelled registrations gracefully.

### Admin area — `src/routes/admin.tsx` (layout)
Layout/guard route for everything under `/admin/*`. A `beforeLoad` checks the session (server-side, via the `getSession` server function) and redirects to `/admin/login` if unauthenticated — but note that's routing UX, not the security boundary; every admin server function independently enforces auth via `authMiddleware` (see [Backend architecture](#backend-architecture) below), so the data is protected even if someone calls the RPC directly.

- **Login — `src/routes/admin.login.tsx`**: `/admin/login`. Email/password form against a real, bcrypt-hashed credential stored in the `admins` table (see `npm run db:seed` output for the demo login).
- **Dashboard — `src/routes/admin.index.tsx`**: `/admin`. Table of all events (newest first) with registration counts, status (Trecut / Full / Deschis), and per-row actions: copy a public share link, view registrants, edit, or delete (with a confirmation modal; delete is a **soft delete** — the event disappears from every list but its registration history is kept).
- **Create/edit event form — `src/routes/admin.events.new.tsx`**: shared `EventForm` used for both `/admin/events/new` and (via `admin.events.$id.edit.tsx`) `/admin/events/$id/edit`. Fields: title, category, location, description, date/start/end time, coach, max seats, optional registration deadline, and a banner image uploader (uploads to the server, 3 MB limit, JPEG/PNG/WEBP/GIF/AVIF/BMP only — formats every major browser can render inline; SVG is rejected as an XSS risk, and HEIC/TIFF are rejected since Chrome/Firefox/Edge can't display them). Warns before shrinking capacity below the current number of registrations.
- **Edit route — `src/routes/admin.events.$id.edit.tsx`**: loads an event by id and renders `EventForm` in edit mode.
- **Registrants — `src/routes/admin.events.$id.registrants.tsx`**: `/admin/events/$id/registrants`. Full roster for one event (name, **decrypted** phone, registration time, active/cancelled status) with a per-row "Anulează" action for staff-initiated cancellation. This is the only place phone numbers are ever returned by the API.

## Backend architecture

Everything lives in this one TanStack Start app — no separate API server. Server-only code lives under `src/server/`; anything that needs to run in the browser imports from `src/lib/store.ts` instead, which re-exports the server functions as TanStack Query `queryOptions()` factories and thin mutation wrappers. TanStack Start's build strips server function bodies out of the client bundle automatically (verified: `dist/client` contains no trace of Prisma, bcrypt, or the encryption key).

- **`src/server/db.ts`** — Prisma client singleton, using the `@prisma/adapter-mariadb` driver adapter (Prisma 7 requires an explicit driver adapter; XAMPP ships MariaDB, hence this adapter rather than a generic MySQL one).
- **`src/server/events.functions.ts`** — `listEvents`, `getEvent` (404s via `notFound()` if missing/soft-deleted), `createEvent`/`updateEvent`/`deleteEvent` (admin-only).
- **`src/server/registrations.functions.ts`** — `createRegistration` (see concurrency section below), `getEventRegistrants` (public, names only), `getRegistrationByToken`/`cancelRegistrationByToken` (self-service), `getEventRegistrationsAdmin`/`adminCancelRegistration` (admin-only, includes decrypted phone).
- **`src/server/auth.functions.ts`** — `login` (rate-limited, bcrypt-verified, constant-time even for unknown emails, rotates the session on success), `logout`, `getSession`.
- **`src/server/uploads.functions.ts`** + **`src/routes/api.uploads.$filename.tsx`** — `uploadBannerImage` (admin-only) validates type/size and writes to `uploads/` on disk; the server route serves files back out by filename. Deliberately factored so swapping to S3-compatible storage later only touches these two files.
- **`src/server/auth-middleware.ts`**, **`csrf-middleware.ts`**, **`rate-limit.ts`**, **`session.ts`**, **`crypto.ts`** — the auth/security primitives described below.

### Concurrency-safe registration

`createRegistration` runs inside a Prisma transaction that issues `SELECT ... FOR UPDATE` on the target event row before counting active registrations and inserting — so two people registering for the last open seat at the same instant can't both succeed. This was verified directly against the real database: firing 25 concurrent registration attempts at a 6-seat event consistently yields exactly 6 successful rows, never more.

### Auth

Sessions are an opaque random token in an `HttpOnly`, `SameSite=Lax` cookie, looked up against a `sessions` table (not a signed JWT — this makes revocation trivial: logout / password change just deletes the row). `authMiddleware` re-checks the session on every admin server function — the `/admin/*` route guard is UX only, per TanStack Start's own security guidance. Login is rate-limited (5/min per IP, in-memory — fine at this app's scale, swap for a shared store if this ever runs across multiple processes) and takes constant time whether or not the email exists, to resist account enumeration.

### Data protection

- **In transit:** the app must be served over HTTPS in production — the session cookie only gets the `Secure` flag when `NODE_ENV=production`, and CSRF's origin check depends on `APP_ORIGIN` matching the real deployed origin.
- **At rest:** registrant `phone` numbers are encrypted with AES-256-GCM (`src/server/crypto.ts`) before being written to MySQL, using `ENCRYPTION_KEY` from the environment. They're only ever decrypted for the admin registrants view — every other query (public roster, event listings) omits the column entirely at the Prisma query level, not just at render time.
- **CSRF:** a global request middleware (`src/server/csrf-middleware.ts`) rejects any non-GET request whose `Origin` header doesn't match `APP_ORIGIN`, as defense-in-depth alongside `SameSite=Lax`.

## Known limitations / next steps

- Rate limiting and the upload directory are process-local — fine for a single-instance deployment (this app's actual scale), but wouldn't survive horizontal scaling without moving to shared storage (Redis, S3).
- No automated tests yet; the concurrency guarantee above was verified with a one-off load script against the real DB rather than a committed test suite.
- No password-reset flow — an admin who forgets their password currently needs a DB update (`UPDATE admins SET passwordHash = ...`) or a re-run of the seed script.
